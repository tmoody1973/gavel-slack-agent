## Session: MOO-52 — Escalation ping (committee → Council)

Same-day second build (after MOO-53), clean per-issue loop: linear-build contract → fresh
worktree off `main` → real-data grounding → written plan → inline TDD (7 tasks) → live
verification gate → PR #24 → Linear In Review. **8 commits, agent 293/293, all green.**

### What we did

A poller-side sibling cron (`0 */6 * * *`) that pings the channels that got a matter's
original alert when its committee **RECOMMENDED FOR ADOPTION/PASSAGE** and it's bound for the
full Common Council's final vote — links back via File #/committee + a Legistar deep-link,
fires once via a new `matterEscalations` Convex ledger.

- **`agent/escalation/detect.js`** — pure `detectEscalation`: latest committee `RECOMMENDED FOR
  ADOPTION/PASSAGE` (result Pass), null if the Council already voted after it.
- **`agent/escalation/sweep.js`** — `runEscalationSweep`, pure over injected I/O: tracked
  matters (sent + matterId, recency-bounded) → detect → re-derive channels via the SAME
  `matchSubscriptions` on the stored detected row → ping → record once.
- **`agent/poller/legistar.js`** — `mapMatter` now exposes `guid/title/status`; `matterDetailUrl`
  builds `LegislationDetail.aspx?ID&GUID` (verified 200).
- **`matterEscalations` Convex ledger** + `detectedItems.listSentWithMatter`. **`escalationCard`**
  bilingual. Cron `escalation-once.mjs` + `escalation-verify.mjs` + crontab line.

### Decisions (asked, user chose)

1. **Trigger = committee recommendation** (≈5 days lead time before the vote), not arrival-at-Council.
2. **Link-back = reference + re-derive channels** via `matchSubscriptions` — no change to the
   proven alert-posting path (it persists neither the channel list nor a message ts).

### The verification gate earned its keep (two real bugs caught live)

The cron dry-run against the real deployment (133 tracked matters) exposed bugs unit tests
never would have:
1. **Already-voted matters were escalating** — e.g. matter 70781 was *adopted June 2*, yet the
   first detector happily pinged "headed to the vote." Fixed: `detectEscalation` returns null
   when a Common Council disposition (ADOPTED/PASSED/PLACED ON FILE/FAILED) follows the
   recommendation. The ping is only meaningful *before* the vote.
2. **Stale recommendations** — matter 65449 had a *2023* "RECOMMENDED FOR ADOPTION" and is now
   being shelved. Fixed with a `recommendedAfter` recency bound (default 21 days). A fresh
   recommendation precedes the vote by days; a 3-year-old one stalled.
   Audited the full real tracked set after both fixes: **133 tracked → 42 genuinely-awaiting,
   0 post-vote leaks.**

### Verification highlights (evidence, not assertions)

- **Positive (real Legistar):** matter 74074 (File #260198, PUBLIC SAFETY & HEALTH recommended
  Jun 11) → exactly 1 ping with the full rendered card (file#, committee, title, heads-up note,
  working Legistar link); re-run with it escalated → 0 (idempotent).
- **Negative (real):** matter 74234 (In Committee, only `ASSIGNED TO`) → `detectEscalation` null.

### Learnings

- **Real `MatterHistory` is the spec.** The committee→Council flow only became designable after
  probing real passed matters: `COMMON COUNCIL ASSIGNED TO` (intro, not an escalation) →
  `[COMMITTEE] RECOMMENDED FOR ADOPTION` → `COMMON COUNCIL ADOPTED` → `MAYOR SIGNED`.
- **A dry-run over the real corpus is worth a hundred unit tests** for a detector — both bugs
  were "plausible and green in tests, wrong on reality."
- Shell single-quote collision (`'Passed'` inside `bash -e '…'`) bit twice; scratch `.mjs` files
  inside `agent/` (for the relative imports) are the reliable probe pattern.

### Risks retired

- Tracked matters no longer reach a final Council vote unannounced.
- `MatterHistory` parsing + the Legistar matter deep-link are proven against real data.

### Next steps

- **Human gate:** eyeball a real escalation ping in Slack.
- **Deploy:** push Convex schema to prod + deploy `gavel-poller`; on first run, suppress the
  ~42-matter catch-up burst (`ESCALATION_REC_MAX_AGE_DAYS` low, or pre-seed the ledger).

## Session: MOO-53 — Watchlist daily sweep (matters + permits)

Clean-context session driven by `docs/HANDOFF-NEXT.md`. Picked MOO-53 over the handoff's

### What we did

The write side (`/gavel watch` / `/gavel unwatch` → Convex `watches`) already existed; the
**sweep** that consumes it did not, so no watch alert had ever fired. Built it as a sibling
daily cron to the Sunday digest (`0 13 * * *`, 8am CT):

- **`agent/watch/match.js`** — pure `matchMatter` (File# exact + case-insensitive substring
  across `MatterFile`/`MatterTitle`/`MatterName`), `classifyEntity` (address vs name, via the
  shared MPROP `normalizeAddress`), `dedupKey`.
- **`agent/poller/legistar.js`** — `fetchRecentMatters(lookbackDays)`: one bulk
  `/matters?$filter=MatterIntroDate ge …` query (data-reference-blessed), newest-first.
- **`agent/watch/sweep.js`** — `runWatchSweep(deps)`, pure orchestration over injected I/O
  (mirrors `poller/poll.js`): match → dedup → group-by-channel → post → record.
- **`agent/convex/watchAlerts.ts` + schema** — dedup ledger keyed `(channelId, entity, kind,
  refId)`, mirroring `detectedAgendaItems`. Stores the caller-stamped `alertedAt` (injected
  clock, so the orchestration stays unit-testable).
- **`agent/blockkit/watch-card.js`** — bilingual `watchCard` (EN framing always; ES framing
  appended for ES channels; file#/address/type stay English per the MOO-43 rule).
- **`mcp-server/src/parcel.js`** — `mapPermit` now surfaces `recordId` + `address` (additive;
  all 47 mcp-server tests still pass) so permits can be deduped + rendered.
- **`agent/scripts/watch-sweep-once.mjs`** (cron) + **`watch-sweep-verify.mjs`** (live proof)
  + crontab line.

### Decisions (asked, user chose)

1. **Next issue = MOO-53** (not MOO-52) — synergy with the just-shipped watch UX.
2. **Unified sweep — matters + permits now**, folding in MOO-68's permit data source rather
   than deferring permits to a separate issue.
3. **Permit matching = full ownership join** (the 0:40 demo beat): address watches hit
   `buildingpermits` directly; LLC/owner watches resolve **owner → MPROP parcels → addresses
   → new permits**.
4. **Matter match fields = File + Title + Name.**

### Surprises / learnings

- **The wired permits dataset (`828e9630…`) has NO owner column** — only Address (+ type/
  status/cost/use/dwelling-units). That's the *reason* LLC→permit needs the MPROP ownership
  join; an LLC can't be matched against the permit rows directly. Confirmed by a live
  one-row CKAN query before designing.
- **The data reference (line 103-107) explicitly blesses the watchlist sweep shape** —
  `/matters?$filter=substringof(...,MatterTitle)&$orderby=MatterIntroDate desc` — so the
  "query new matters citywide" approach beat "reuse the agenda stream" cleanly.
- **Convex validator vs injected clock:** first verify run failed because `runWatchSweep`
  passes `alertedAt` but the mutation validator only had the 4-field key + re-stamped
  server-side. Fixed by widening the insert validator (`alertRecord`) to store the
  caller's `alertedAt` — keeps the unit test's deterministic clock honest.
- **Verify-script term derivation bug:** first attempt picked two non-adjacent title words
  ("Presentation discussion" from "Presentation **and** discussion…") → no substring match.
  Switched to a single distinctive word. Good reminder that the *test harness* can be wrong
  while the unit under test is right.
