## TL;DR

No code this session — an architecture call. Tarik stood up `mke-alerts@agentmail.to` (an AgentMail inbox) subscribed to the City/County **E-Notify** firehose and asked how to turn those emails into custom alerts + how to store them for good historical search. Verdict: it's a high-leverage **second source adapter** into the pipeline we already built, store it **hybrid in Convex (not a vector store)**, parse PDFs by handing them to Claude natively, and **park it behind the spine**. Captured as `docs/agentmail-civic-ingestion.md` + Linear **MOO-69**.

## What we did
- Grounded the answer in real **AgentMail docs** (Context7): inbound via WebSocket `message.received` or webhook; message fields incl. `extracted_text`, `attachments`, `labels`; `getAttachment(inbox, msgId, attId)` returns attachment bytes; `list/get` for backfill.
- Reframed AgentMail as a **second heartbeat** alongside the Legistar poller — broader civic surface (per-alder news, police/crime bulletins, property sales, development plans, transit/health alerts) that the Legistar *API* doesn't expose as alert-ready — feeding the **same** downstream (extract → match subscriptions → summarize → alert).
- Wrote the design doc and filed **MOO-69** (Gavel project, Backlog, Medium, **blocked by MOO-44**, doc linked) with a full Intent/Acceptance/Verification/Out-of-scope contract.

## Decisions made
1. **Hybrid Convex storage, not vector-first.** `civicNotifications` table: indexed fields (district/category/fileNumber/date) + full-text search on subject+body + *optional* vector index for "find similar". One store, no Pinecone/Weaviate. Rationale: civic search is dominated by temporal + categorical + entity-exact filters where relational beats vectors; semantic is the minority case. These are **records, not RAG passages** (different shape from the `zoning_code`/`transcripts` namespaces).
2. **PDF attachments → Claude document blocks**, not a `pdf-parse` library. Sonnet 4.6 reads PDFs natively (handles scanned/messy civic layouts), reuses the existing Claude call, zero new deps. Page/size cap; only pulled when the email body is thin (summarizer fallback chain).
3. **Reuse the spine, don't fork.** E-Notify categories map onto MOO-45 subscriptions (committees/keywords/district); `subject→title`, `extracted_text→matterText` map onto the MOO-42 summarizer (generalize "matter" → "civic item"); MOO-44 renders the card.
4. **Fusion/dedup vs Legistar** by extracted fileNumber/address — don't double-alert across the two heartbeats.
5. **Sequencing:** park behind the spine. First card stays **Legistar-sourced** (depth: hearing date + alderperson + "how to be heard" — the demo's credibility anchor); AgentMail is the breadth multiplier right after. Tarik chose "track it + capture the design," explicitly opening MOO-69 despite the scope freeze (tracked, not built — spirit intact).

## What I learned (Tarik)
- **Compliance reframe:** E-Notify emails are official government broadcasts = **public record**, so storing/indexing them is the *intended* "index the public record" path — categorically different from the Slack-message RTS-only rule, and not the mem0-style private persistence the PRD rejected. The guardrail is keeping that inbox civic-only.
- **"Vector store" is a reflex, not always the answer.** For metadata-rich, time-series notifications, filter-first relational + full-text wins; embeddings are a bonus modality. Match the store to the query shapes, not to the hype.
- **Adding a source ≠ adding a product** when the downstream pipeline is already built — AgentMail is ~an adapter.

# Session 2 — MOO-41: the Legistar poller (the heartbeat) ✅

**Focus:** Build MOO-41 — a scheduled poller that detects new `Final` agenda items and enqueues summarize+alert jobs, idempotently. Worktree `worktree-moo-41-legistar-poller`; branch-per-issue, merges to main.

## TL;DR
Built the spine's heartbeat end-to-end with TDD: a pure idempotent **diff core**, an injected **Legistar OData client**, a **poll orchestrator**, a Convex **`detectedAgendaItems` ledger+queue**, a live verify harness, and a **Fly.io `*/5` cron** (supercronic). **63/63 tests**, verified against **real Milwaukee Legistar + live Convex**, deployed to Fly app `gavel-poller`.

## What we did (per the proven MOO-42/45 pattern)
- **Design decisions locked up front** (2 load-bearing, surfaced via AskUserQuestion): dedup key = **`EventItemId`** (`client:eventItemId`); enqueue seam = **one `detectedAgendaItems` table** that is both idempotency ledger and alert queue (`alertStatus pending|sent`), drained by MOO-44. - Plan written first: `docs/superpowers/plans/2026-06-08-moo-41-legistar-poller.md`.

## Two real-data findings the verify gate caught (not papered over)
1. **Boilerplate flood.** Live data: **89 of 179** agenda items carry an `EventItemMatterId`; the other 90 are meeting headers/webcast/accessibility notices. Decision (Tarik): **enqueue only matter-bearing items** — a "summarize+alert job" presupposes a matter. Added as a filter in `runPoll` (TDD'd), cutting the queue to the 89 real matters (ordinances, zoning changes, resolutions).
2. **`EventAgendaLastPublishedUTC` has no `Z`.** Raw value `2026-06-08T14:38:48.597` is genuinely UTC but unsuffixed, so `new Date()` misparsed it as local Central → 3h future → bogus **negative latency**. Fixed at the boundary: `mapEvent` tags it `Z` (only the instant, not `eventDate` which is a local calendar date). Latency then read a sane **+120 min** backfill age.

## Evidence (real, not asserted)
- `node scripts/poller-verify.mjs` → RUN 1 (cold): fetched 89, **detected 89 NEW**; RUN 2 (immediate): fetched 89, **detected 0** → `IDEMPOTENT ✓`.

- Latency criterion: structurally **< 20 min** via the 5-min cron (a new Final agenda is caught on the next tick); the backfill "120 min" is just the age of pre-existing agendas, not cron lag.
- Fly deploy succeeded (remote builder, 385 MB image; supercronic verified checksum `71b0d58…`). Cron ticks: _(see Linear evidence comment)_.

## Decisions / learnings (Tarik)
- **`{client}`-aware throughout** (`milwaukee` | `milwaukeecounty`) — County (MOO-66) is a one-line flip; the dedup key is client-scoped so the same EventItemId across cities stays distinct.
- **One table, two jobs (ledger + queue)** beat two tables: a row's existence = idempotency, its `alertStatus` = work state. Cohesive, durable across Fly restarts, trivially inspectable.
- **Verify against reality surfaces the real spec.** Neither the boilerplate split nor the timezone bug was visible from the docs — only live data showed them. That's the whole point of the gate.

# Session 5 — Gavel goes interactive: deploy the Bolt app + give it a face (MOO-70, MOO-71) ✅

## TL;DR
Took Gavel from "poller posts alerts" to a **fully hosted, interactive civic agent**: deployed the Bolt app to Fly (Socket Mode), gave it the Gavel bot identity, replaced the scaffold persona with a real Milwaukee civic identity, and fixed a latent MCP bug that was breaking every tool call. Gavel now answers real civic questions in Slack, in English and Spanish, from live Legistar data — no laptop required.

## MOO-70 — deploy the interactive app
- The interactive surface (buttons, DMs, MCP tools) only ran via `slack run` ("Gavel (local)"); the buttons showed a ⚠️ because nothing was listening when no laptop session was open. Two separate programs: the **poller** (deployed) and the **interactive app** (wasn't).
- Token gate (retired first, curl-before-commit): the app needs static `xoxb` (bot) + `xapp` (app-level) tokens for Socket Mode. The CLI only stores a rotating `xoxe` refresh token, so they had to be generated from the app config page — no admin needed (app already installed). Proved by booting `app.js` standalone → `Gavel is running!`.
- New Fly app `gavel-app`: `Dockerfile.app` + `fly.app.toml`, repo-root build context bundling `agent/` + `mcp-server/`, no HTTP port (Socket Mode dials out). Poller switched to the bot token (one secret — it already preferred `SLACK_BOT_TOKEN`).
- **Root-cause bug #1:** every DM crashed with "Claude Code process exited with code 1." The agent runs on `@anthropic-ai/claude-agent-sdk`, which *spawns the Claude Code CLI* — and Claude Code refuses `--dangerously-skip-permissions` (the agent's `bypassPermissions`) **as root**. Fix: run the container as the non-root `node` user. (Diagnosed by reproducing the binary in-container: `claude --version` fine, real key fine, but bypass-as-root → refusal.)

## MOO-71 — give Gavel its civic identity
- The deployed agent literally introduced itself as "a friendly Slack assistant" — the `bolt-js-starter-agent` scaffold. Civic suggested prompts + App Home; dropped the forced emoji-reaction gimmick and the confusing "Slack MCP disconnected" widget.
- **Root-cause bug #2 (the big one):** Gavel answered the first civic question with a "technical error." The milwaukee-civic tools returned an **array** as `structuredContent`, which MCP rejects with **-32602** — so every list tool failed *when the agent called it over the protocol*. The MOO-47 unit tests used a fake server and never exercised real MCP; `mcp-verify.mjs` called the client directly; both missed it. Fix: drop `structuredContent` (the text payload already carries the JSON) + add a real MCP-client stdio integration test (`mcp-protocol.test.js`) that reproduces it.
